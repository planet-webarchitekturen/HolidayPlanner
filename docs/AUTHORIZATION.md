# Authorization — Holiday Planner

## Approach: Symmetric JWT with Shared Secret

All services use **HS384 JSON Web Tokens** signed with a shared HMAC secret. The identity-service issues tokens; every other service validates them independently using the same key.

No external identity provider (Keycloak, Auth0, etc.) is used — the symmetric approach keeps the stack simple and avoids an additional runtime dependency.

---

## Token Generation (identity-service)

**Login endpoint:** `POST /api/identity/auth/login` (also available at `/api/auth/login`)

```json
Request:  { "email": "user@example.com", "password": "..." }
Response: { "token": "eyJ...", "userId": "uuid", "email": "...",
            "organizationId": "uuid", "role": "USER" }
```

The token is generated by `JwtTokenProvider.generateToken()` in identity-service using jjwt 0.12.3.

---

## JWT Claims

| Claim | Type | Description |
|---|---|---|
| `sub` | UUID string | User ID |
| `organizationId` | UUID string | Organization the user belongs to |
| `roles` | List\<String\> | e.g. `["USER"]`, `["ACCOUNTANT"]` |
| `email` | String | User email |
| `iat` | Timestamp | Issued-at |
| `exp` | Timestamp | Expiry (24 hours from issue) |

**Secret:** `${JWT_SECRET}`, defaulting to `holidayplanner-super-secret-key-that-is-at-least-256-bits-long`
**Algorithm:** HS384 (auto-selected by jjwt from key length)
**Expiry:** 86 400 000 ms (24 hours)

## Token Validation in Each Service

All services include the `shared` module which provides `JwtAuthenticationFilter` (a Spring Security `OncePerRequestFilter`). Each service's `SecurityConfig` registers this filter:

```java
@Bean
public JwtAuthenticationFilter jwtAuthenticationFilter(
        @Value("${jwt.secret}") String secret) {
    return new JwtAuthenticationFilter(secret);
}
```

The filter:
1. Reads the `Authorization: Bearer <token>` header
2. Validates the signature and expiry with the shared secret
3. Extracts `userId`, `organizationId`, `roles`, `email` from claims
4. Populates `SecurityContextHolder` with a `UsernamePasswordAuthenticationToken` whose principal is a `JwtClaims` object
5. Returns **401 Unauthorized** for missing or invalid tokens
6. Passes through for public endpoints (health, login, register)

**Helper utilities** in `com.holidayplanner.shared.security`:
- `JwtClaims` — typed holder for extracted claims
- `SecurityUtils.getCurrentUser()` — retrieves `JwtClaims` from context
- `SecurityUtils.getCurrentOrganizationId()` — convenience accessor

## Role-Based Access Control

Roles are stored as enum `UserRole` in the shared model and embedded in the JWT.

| Role | Meaning |
|---|---|
| `USER` | Standard registered user |
| `EVENT_OWNER` | Can create / update events and terms |
| `ORGANIZATION_TEAM_MEMBER` | Organization staff |
| `ACCOUNTANT` | Can mark payments paid / refund |
| `ADMIN` | Can create organizations |

### booking-service
| Endpoint | Required Role |
|---|---|
| `POST /api/bookings` | USER, EVENT_OWNER, ORGANIZATION_TEAM_MEMBER |
| `DELETE /api/bookings/{id}` | EVENT_OWNER, ORGANIZATION_TEAM_MEMBER, ADMIN |
| All GET endpoints | Any authenticated user |

### payment-service
| Endpoint | Required Role |
|---|---|
| `PATCH /api/payments/{id}/pay` | ACCOUNTANT |
| `PATCH /api/payments/{id}/refund` | ACCOUNTANT |
| Other endpoints | Any authenticated user |

### event-service
| Endpoint | Required Role |
|---|---|
| `POST /api/events` | EVENT_OWNER, ORGANIZATION_TEAM_MEMBER, ADMIN |
| `PUT /api/events/{id}` | EVENT_OWNER, ORGANIZATION_TEAM_MEMBER |
| Other endpoints | Any authenticated user |

### organization-service
| Endpoint | Required Role |
|---|---|
| `POST /api/organizations` | ADMIN |
| `POST /api/organizations/{id}/team-members` | ORGANIZATION_TEAM_MEMBER, ADMIN |
| Other endpoints | Any authenticated user |

### identity-service
| Endpoint | Required Role |
|---|---|
| `GET /api/identity/users/{id}` | Any authenticated user |

## Organization-Scoped Access

The `organizationId` claim in the JWT restricts what data a user can access. Services can retrieve the current user's organization via:

```java
UUID orgId = SecurityUtils.getCurrentOrganizationId();
```

Example use case: booking-service verifies the event term belongs to the user's organization before creating a booking.

---

## IPC — Inter-Service JWT Forwarding

When booking-service calls event-service to validate an event term, it forwards the user's JWT:

```java
// EventServiceClient.java (booking-service)
headers.setBearerAuth(extractCurrentToken());
```

`extractCurrentToken()` reads the `Authorization` header from the current `HttpServletRequest` via `RequestContextHolder`. This ensures downstream services apply the same auth checks.

## Configuration

### application.yml (all services)
```yaml
jwt:
  secret: ${JWT_SECRET:holidayplanner-super-secret-key-that-is-at-least-256-bits-long}
```

### docker-compose.yml
```yaml
environment:
  JWT_SECRET: ${JWT_SECRET:-holidayplanner-super-secret-key-that-is-at-least-256-bits-long}
```

The secret must be **identical** across all services. In production, set the `JWT_SECRET` environment variable to a cryptographically random 32+ byte value.

## Open Questions

1. **Token refresh** — There is currently no refresh-token mechanism. Users must re-login after 24 hours.
2. **Token revocation** — JWTs cannot be revoked before expiry. A token blacklist or short-lived tokens would be needed for logout.
3. **Role assignment** — Roles are set at registration time and cannot currently be changed via the API. An admin endpoint to update roles is not yet implemented.
4. **Secret rotation** — Rotating the shared secret requires a coordinated deployment of all services simultaneously.
5. **HTTPS** — The stack currently uses plain HTTP. All JWT traffic should be encrypted in production.
